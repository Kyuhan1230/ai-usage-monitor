# Compact 창 반응형 UI 개발 계획

- 상태: 구현·검증 완료 (2026-07-22)
- 목표 버전: `1.2.2`
- 대상 화면: `src/ui/compact.html`, `src/ui/compact.css`
- 창 생성 코드: `src-tauri/src/lib.rs`

## 1. 배경

Compact 창은 현재 기본 `360 x 480`, 최소 `280 x 320` 논리 픽셀로 생성된다. 기본 너비에서는 Codex와 Claude 카드가 세로로 배치되며, 제목·카드 2개·결정 안내·설정·하단 버튼의 자연 높이가 기본 창 높이를 넘는다. 그 결과 사용자가 창을 줄이지 않았는데도 최초 표시부터 세로 스크롤이 나타난다.

현재 `max-height: 560px` 규칙은 패딩과 간격만 일부 줄인다. 본문, 제목, 상태 문구와 버튼의 글꼴 및 행 높이는 대부분 유지되므로 `360 x 480`에 콘텐츠를 맞추기에는 부족하다. 또한 결정 안내의 본문에 `overflow: auto`가 적용되어 창 전체 스크롤과 별개의 내부 스크롤이 생길 가능성이 있다.

## 2. 목표

1. 기본 `360 x 480` 창에서는 로딩·정상·경고 상태 모두 최초 세로 스크롤이 없어야 한다.
2. `320 x 480`에서도 가로 스크롤, 요소 겹침과 잘린 조작 요소가 없어야 한다.
3. 최소 `280 x 320`에서는 핵심 정보와 모든 기능에 접근할 수 있어야 하며, 필요한 경우 창 전체의 단일 세로 스크롤만 허용한다.
4. 창 크기가 커지면 글꼴·간격·패딩이 원래 밀도로 자연스럽게 복원되어야 한다.
5. 핵심 잔여량 수치와 버튼의 가독성 및 클릭 가능 크기를 과도하게 축소하지 않는다.

## 3. 비목표

- `transform: scale()` 또는 CSS `zoom`으로 화면 전체를 축소하지 않는다.
- 정보 영역을 임의로 제거하거나 기능을 숨기지 않는다.
- Compact 창의 기본 크기 `360 x 480`과 최소 크기 `280 x 320`은 우선 유지한다.
- 이번 변경에서 Insights, Details, Setup, Update 창의 디자인을 함께 변경하지 않는다.

## 4. 구현 원칙

### 4.1 반응형 밀도 토큰

`compact.css`의 글꼴, 패딩, 간격과 최소 높이를 CSS 사용자 정의 속성으로 정리한다. 기본값은 현재 가독성을 유지하고, 높이 기반 미디어 쿼리에서 토큰만 덮어써서 규칙이 여러 선택자에 분산되지 않도록 한다.

예상 토큰은 다음과 같다.

```css
:root {
  --compact-font-body: 13px;
  --compact-font-title: 18px;
  --compact-font-meta: 12px;
  --compact-shell-padding: 14px;
  --compact-shell-gap: 10px;
  --compact-card-padding: 12px;
  --compact-control-height: 30px;
}
```

### 4.2 높이 기반 밀도 단계

첫 단계는 `max-height: 520px`을 기준으로 적용한다. 실제 구현 전후 측정 결과에 따라 경계값은 소폭 조정할 수 있지만, 기본 `360 x 480`이 이 단계에 포함되어야 한다.

권장 축소 범위는 다음과 같다.

| 항목 | 기본 | 낮은 창 |
| --- | ---: | ---: |
| 본문 글꼴 | 13px | 12px |
| 제목 글꼴 | 18px | 16px |
| 상태·리셋·시간 문구 | 12px | 11px |
| 버튼 최소 높이 | 30px | 28px |
| 결정 안내 최소 높이 | 58px | 약 50px |
| Shell 패딩 | 14px | 8~10px |
| Shell 행 간격 | 10px | 6~8px |
| 카드 세로 패딩 | 12px | 7~8px |

글꼴은 `11px` 미만으로 줄이지 않는다. 잔여량 숫자는 보조 문구보다 높은 시각적 우선순위를 유지한다.

### 4.3 너비와 높이 규칙의 결합

기존 `max-width: 340px` 규칙은 유지하되 낮은 높이 규칙과 동시에 적용되는 경우를 검증한다.

- 제한 수치와 속도 문구가 겹치지 않아야 한다.
- 리셋 시각과 갱신 시각은 한 줄 말줄임을 기본으로 한다.
- 불가피하게 줄바꿈되는 경우에도 가로 스크롤은 만들지 않는다.
- 하단 버튼은 최소 너비에서 2열 배치를 유지하고 잘리지 않아야 한다.

### 4.4 스크롤 정책

- `body` 또는 `.shell` 중 한 곳만 주 세로 스크롤 컨테이너로 사용한다.
- 가로 스크롤은 항상 차단한다.
- `.decision-strip strong`의 `overflow: auto`는 `overflow: hidden`으로 변경하여 중첩 스크롤을 제거한다.
- `360 x 480`에서는 `scrollHeight <= clientHeight`를 만족해야 한다.
- 최소 `280 x 320`에서 세로 스크롤이 필요하면 허용하되 모든 조작 요소까지 도달할 수 있어야 한다.

## 5. 구현 순서

1. 로딩·정상·경고 상태에서 현재 콘텐츠 높이와 스크롤 발생 위치를 기록한다.
2. 글꼴·패딩·간격·버튼 높이를 반응형 밀도 토큰으로 정리한다.
3. `max-height` 밀도 단계를 추가하고 기본 `360 x 480`에 맞춘다.
4. 기존 `max-width: 340px` 규칙과 결합했을 때의 줄바꿈 및 버튼 배치를 조정한다.
5. 결정 안내 내부 스크롤을 제거하고 창 전체 단일 스크롤 정책을 적용한다.
6. 대표 크기와 상태별 스크린샷을 비교하고 과도하게 작은 글꼴이나 잘린 콘텐츠를 수정한다.
7. UI 계약 테스트와 전체 회귀 테스트를 실행한다.
8. 비평 subagent 검토 결과를 반영한 뒤 릴리스 준비를 진행한다.

## 6. 검증 매트릭스

| 창 크기 | 로딩 | 정상 | 경고·긴 문구 | 기대 결과 |
| --- | --- | --- | --- | --- |
| `680 x 560` | 확인 | 확인 | 확인 | 넓은 2열 카드, 기본 밀도 유지 |
| `360 x 560` | 확인 | 확인 | 확인 | 세로 카드, 스크롤 없음 |
| `360 x 480` | 필수 | 필수 | 필수 | 최초 세로·가로 스크롤 없음 |
| `320 x 480` | 필수 | 필수 | 필수 | 가로 스크롤과 겹침 없음 |
| `280 x 320` | 필수 | 필수 | 필수 | 단일 세로 스크롤 허용, 모든 기능 접근 가능 |

추가로 Windows 디스플레이 배율 100%와 125%에서 기본 창을 확인한다. 가능하면 150%에서도 최소 창이 화면 작업 영역을 벗어나지 않는지 점검한다.

## 7. 테스트 계획

### 자동 테스트

- `tests/ui-tests.js`에 Compact 높이 반응형 규칙의 존재를 검증하는 계약을 추가한다.
- 가로 스크롤 차단, 최소 글꼴, 결정 안내 중첩 스크롤 방지 규칙을 검증한다.
- 기존 창 작업 영역 맞춤 테스트와 Rust updater 테스트를 그대로 통과시킨다.
- 최종 게이트는 `npm test`로 실행한다.

### 시각 검증

- 검증 매트릭스의 크기별 스크린샷을 남긴다.
- 로딩 placeholder와 실제 사용량 데이터 양쪽을 확인한다.
- 경고 문구가 두 줄일 때 결정 안내 내부에 별도 스크롤바가 생기지 않는지 확인한다.
- 창을 연속으로 축소·확대해 미디어 쿼리 경계에서 레이아웃이 튀거나 겹치지 않는지 확인한다.

## 8. 완료 조건

- 기본 `360 x 480`의 모든 대표 상태에서 세로·가로 스크롤이 없다.
- `320 x 480`에서 정보와 버튼이 겹치지 않는다.
- `280 x 320`에서 가로 스크롤이 없고 모든 조작 요소까지 접근할 수 있다.
- 보조 글꼴은 최소 `11px`, 버튼 높이는 최소 `28px`을 유지한다.
- 중첩 스크롤 컨테이너가 없다.
- 전체 테스트와 Windows 설치 빌드가 통과한다.
- 변경 전후 스크린샷과 비평 결과가 구현 커밋 또는 PR에 첨부된다.

## 9. 커밋 및 릴리스 계획

구현과 검증은 다음 단위로 나눈다.

```text
fix(ui): make compact window height-responsive
test(ui): cover compact responsive layout
chore(release): prepare v1.2.2
```

첫 번째 커밋에는 Compact CSS와 필요한 최소 마크업 변경만 포함한다. 두 번째 커밋에는 회귀 테스트와 검증 보조 코드만 포함한다. 마지막 커밋에서 버전 선언, 잠금 파일과 변경 이력을 `1.2.2`로 맞춘다.

PR CI와 Windows 설치 빌드가 통과한 뒤 `main` 병합 커밋에 `v1.2.2` 태그를 붙인다. Release 워크플로가 설치 파일, Tauri updater 서명과 `latest.json`을 모두 검증하고 공개한 뒤 실제 최신 업데이트 응답이 `1.2.2`인지 스모크 테스트한다.

| 긴 동적 문구가 예상보다 많은 높이를 차지함 | 대표 최장 문구 상태를 검증하고 보조 문구는 말줄임 적용 |
