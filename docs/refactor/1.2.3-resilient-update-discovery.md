# 1.2.3 놓치지 않는 업데이트 발견

작성일: 2026-07-23 (Asia/Seoul)  
상태: 구현 전 확정 설계  
대상 버전: `1.2.3`

## 한 줄 결론

업데이트 확인을 앱 시작 시 한 번만 실행하는 구조에서 앱이 켜진 동안 계속 다음 확인 시각을 예약하는 구조로 바꾼다. 새 버전을 발견하면 Windows 알림으로 한 번 안내하고, 트레이 메뉴에 `v1.2.3 업데이트 가능`을 설치할 때까지 유지한다. 업데이트 창은 사용자가 트레이 또는 설정 화면에서 업데이트를 열었을 때만 표시한다.

## 배경

`1.2.2`의 자동 업데이트 확인은 앱 시작 15초 뒤 한 번만 실행된다. 마지막 성공 확인 후 24시간이 지나지 않았으면 `skipped`를 반환하고 작업이 끝난다.

실제 확인 사례:

- 앱 시작: 2026-07-23 08:10 KST
- 마지막 성공 확인: 2026-07-22 11:56 KST
- 시작 시 경과 시간: 약 20.6시간
- 결과: 시작 15초 뒤 확인이 생략됐고, 앱이 계속 실행돼도 11:56에 다시 확인하지 않음

설치된 `1.2.2`가 당시 최신 버전이므로 업데이트 창이 뜨지 않은 것 자체는 정상이다. 문제는 사용자가 정상적인 무응답과 확인 누락을 구분할 수 없고, 트레이에서 오래 실행되는 앱이 새 릴리스를 다음 시작 또는 수동 확인 전까지 발견하지 못할 수 있다는 점이다.

현재 구조의 근거:

- `src-tauri/src/lib.rs`: `start_automatic_update_check`가 15초 뒤 `check_and_show_update`를 한 번만 호출함
- `src-tauri/src/update.rs`: `lastSuccessfulCheckAt + 24시간` 이내면 자동 확인을 생략함
- `src-tauri/src/lib.rs`: 자동 확인 결과와 오류를 `let _ = ...`로 버림
- `src-tauri/src/lib.rs`: 트레이의 `업데이트 확인`은 수동 확인을 실행함
- `~/.codex-usage-wrapper/update-state.json`: 확인 시각과 알림·미루기 상태를 앱 버전과 무관하게 유지함

## 목표

1. 앱을 재시작하지 않아도 24시간 제한이 끝난 시점에 자동 확인한다.
2. 앱 실행 중 새 릴리스가 공개돼도 늦어도 다음 자동 확인에서 발견한다.
3. 앱 버전이 바뀐 뒤에는 이전 버전의 확인 시각과 무관하게 한 번 확인한다.
4. 새 버전을 발견하면 포커스를 빼앗지 않고 Windows 알림과 트레이 상태로 알린다.
5. Windows 알림이 차단돼도 트레이에서 업데이트 가능 상태를 계속 확인할 수 있게 한다.
6. 자동 확인 실패는 반복 팝업 없이 재시도하고, 설정 화면에서 마지막 결과를 확인할 수 있게 한다.
7. 다운로드와 설치는 계속 사용자 승인 뒤에만 시작한다.

## 하지 않는 일

- 사용자 승인 없는 다운로드·설치
- 새 버전 발견 즉시 업데이트 창을 강제로 앞으로 가져오기
- 매분 GitHub에 네트워크 요청하기
- 범용 HTTP 클라이언트나 `tauri-plugin-http` 추가
- 프런트엔드에 updater plugin 권한 직접 노출
- 사용량 기록, 분석 파일 또는 CLI 인증 데이터 변경
- Windows 알림 클릭으로 업데이트 창 열기

마지막 항목은 현재 고정된 `tauri-plugin-notification 2.3.3`의 데스크톱 Rust API가 알림 표시만 제공하고 활성화 콜백을 제공하지 않기 때문이다. 알림 클릭은 이번 버전의 완료 조건에 넣지 않는다. 사용자가 확실히 이용할 수 있는 진입점은 트레이 메뉴와 설정 화면이다.

## 사용자 경험 계약

### 상태별 동작

| 상황 | Windows 알림 | 트레이 메뉴 | 업데이트 창 | 설정 화면 |
| --- | --- | --- | --- | --- |
| 아직 확인하지 않음 | 없음 | `업데이트 확인` | 없음 | `아직 확인하지 않음` |
| 최신 버전 | 없음 | `업데이트 확인` | 없음 | `최신 · 마지막 확인 시각` |
| 새 버전 자동 발견 | 버전당 한 번 | `vX.Y.Z 업데이트 가능` | 자동으로 열지 않음 | 새 버전과 마지막 확인 시각 |
| 새 버전 수동 발견 | 선택적 성공 알림 없음 | `vX.Y.Z 업데이트 가능` | 즉시 열기 | 새 버전 표시 |
| 자동 확인 실패 | 없음 | 기존 상태 유지 | 없음 | `마지막 확인 실패 · 다시 시도` |
| 수동 확인 실패 | 기존 방식대로 알림 또는 화면 오류 | 기존 상태 유지 | 없음 | 재시도 가능한 오류 |
| 설치 중 | 없음 | `vX.Y.Z 업데이트 가능` 유지 | 진행률 표시 | 기존 상태 유지 |

### 트레이 동작

- 기존 `check_update` 메뉴 ID를 유지한다.
- 기본 문구는 `업데이트 확인`이다.
- 새 버전을 알면 문구를 `vX.Y.Z 업데이트 가능`으로 바꾼다.
- 해당 항목을 누르면 수동 재확인 후 기존 업데이트 창을 연다.
- 이미 알고 있는 버전의 확인이 일시적으로 실패해도 `업데이트 가능` 문구를 지우지 않는다.
- 최신 버전 확인 또는 설치 후에는 `업데이트 확인`으로 되돌린다.
- 고정된 `tauri 2.11.5`의 `MenuItem::set_text`를 사용하므로 트레이 전체를 다시 만들지 않는다.

### 알림과 미루기

- 자동 발견 알림 제목: `새 버전이 있습니다`
- 자동 발견 알림 본문: `vX.Y.Z 업데이트를 사용할 수 있습니다. 트레이 메뉴에서 확인하세요.`
- 같은 버전은 프로세스를 재시작해도 한 번만 알린다.
- 더 높은 버전은 이전 버전의 알림·미루기 상태와 무관하게 새로 알린다.
- `나중에`와 업데이트 창 닫기(X)는 같은 버전의 추가 Windows 알림을 24시간 억제한다.
- 미룬 동안에도 트레이의 `업데이트 가능` 문구는 유지한다. 사용자가 직접 누르면 언제든 창을 열 수 있다.

### 최신 버전일 때

자동 확인 성공을 Windows 알림으로 매번 알리지 않는다. 정상 상태의 반복 알림은 소음이므로 설정 화면의 마지막 확인 시각만 갱신한다.

## 스케줄러 계약

### 시작 규칙

1. 앱 시작 직후 네트워크를 사용하지 않는다.
2. 최소 15초를 기다린다.
3. 현재 앱 버전과 `lastSuccessfulCheckAppVersion`이 다르면 이전 24시간 제한을 무시하고 확인한다.
4. 같은 버전이고 마지막 성공 확인 후 24시간이 지나지 않았다면 남은 시간까지 예약한다. 작업을 종료하지 않는다.
5. 성공 기록이 없으면 15초 뒤 확인한다.

### 실행 중 규칙

- 성공해서 최신 또는 새 버전을 확인하면 다음 자동 확인은 성공 시각부터 24시간 뒤다.
- 실패하면 15분, 1시간, 6시간 순서로 재시도하고 이후에는 6시간 간격을 유지한다.
- 성공하면 자동 실패 횟수를 0으로 되돌린다.
- 수동 확인 실패는 자동 실패 횟수를 늘리지 않는다.
- 수동 확인 성공은 마지막 성공 시각을 갱신한다. 스케줄러가 기존 시각에 일찍 깨어나더라도 상태를 다시 읽고 다음 시각만 재계산한다.
- 다른 확인이 진행 중이면 실패로 기록하지 않고 1분 뒤 다시 판단한다.
- 시스템 절전과 시각 변경에 대응하도록 한 번의 대기 시간은 최대 1시간으로 제한하고, 깨어날 때마다 상태 파일과 현재 시각을 다시 읽는다.
- 네트워크 요청은 `due`일 때만 실행한다. 매시간 깨어나는 것은 로컬 시각 계산이며 GitHub 요청이 아니다.

### 구현 방식

기존 활동 감시와 같은 전용 `std::thread`를 하나 사용한다. 이 스레드는 대부분 잠들어 있으며, 확인이 필요할 때만 Tauri async runtime에서 updater 확인 future를 실행한다. 장시간 `spawn_blocking` 작업으로 async blocking pool을 점유하지 않고, 새 타이머 의존성도 추가하지 않는다.

개략 흐름:

```text
앱 시작
  -> 15초 대기
  -> loop
       -> 상태·현재 앱 버전·현재 시각 읽기
       -> next_update_check_decision 계산
       -> Wait: 최대 1시간 대기 후 다시 계산
       -> Busy: 1분 뒤 다시 계산
       -> Check: updater 확인
            -> 최신: 상태 저장, 트레이 기본값
            -> 새 버전: 상태 저장, 트레이 강조, 필요하면 알림
            -> 실패: 상태 저장, backoff 예약
```

## 확인 출처와 표시 정책

외부 command의 `manual: bool` 호환성을 유지하고 내부도 두 출처만 구분한다.

```text
UpdateCheckSource
  Automatic
  Manual
```

확인과 표시를 분리한다.

```text
run_update_check(source) -> UpdateCheckResult
update_effects(source, result, state) -> CheckEffects
apply_update_effects(app, effects)
```

`CheckEffects`는 다음 두 값만 결정한다.

- Windows 알림을 보낼 버전
- 업데이트 창 표시 여부

트레이 문구와 설정 화면 상태는 별도 표시 모델로 저장하지 않고 `availableVersion`, 현재 앱 버전, 마지막 성공 시각과 마지막 오류에서 바로 계산한다.

업데이트 다운로드 URL, 서명 검증과 설치는 기존 Rust updater 경로가 계속 소유한다.

## 상태 파일 v2

`~/.codex-usage-wrapper/update-state.json`을 다음 형태로 확장한다.

```json
{
  "schemaVersion": 2,
  "lastSuccessfulCheckAt": "2026-07-23T12:00:00+09:00",
  "lastSuccessfulCheckAppVersion": "1.2.3",
  "lastAutomaticAttemptAt": "2026-07-23T12:00:00+09:00",
  "consecutiveAutomaticFailures": 0,
  "lastCheckError": null,
  "availableVersion": "1.2.4",
  "lastNotifiedVersion": "1.2.4",
  "snoozeUntil": null
}
```

필드 계약:

- `lastSuccessfulCheckAt`: 최신 또는 새 버전 응답을 정상적으로 받은 시각
- `lastSuccessfulCheckAppVersion`: 위 성공 확인을 수행한 앱 버전
- `lastAutomaticAttemptAt`: 자동 재시도 간격 계산용 시각
- `consecutiveAutomaticFailures`: 자동 실패 backoff 단계
- `lastCheckError`: 설정 화면용 짧은 오류. 줄바꿈을 제거하고 300자로 제한하며 인증값이나 사용자 파일 내용을 넣지 않음
- `availableVersion`: 마지막으로 정상 확인한 더 높은 버전. 최신 확인 또는 설치 뒤 제거
- `lastNotifiedVersion`: Windows 알림을 마지막으로 표시한 버전
- `snoozeUntil`: 기존 업데이트 창의 `나중에` 계약과 v1 JSON 호환을 위해 유지하는 시각. 알림 1회 정책과 자동 창 제거 이후에는 스케줄러 핵심 상태로 사용하지 않음

### 마이그레이션

- 기존 v1 JSON은 `#[serde(default)]`로 그대로 읽는다.
- 없는 필드는 기본값을 사용하고 다음 저장 때 `schemaVersion: 2`로 기록한다.
- 기존 `lastSuccessfulCheckAt`은 보존한다.
- v1에는 확인한 앱 버전이 없으므로 `lastSuccessfulCheckAppVersion`은 비워 둔다. 따라서 v2 앱 첫 시작에는 15초 뒤 한 번 확인한다.
- 손상된 파일은 현재와 같이 기본 상태로 복구하며 앱 시작을 막지 않는다.
- 다운로드 경로, 설치 파일, 서명키와 릴리스 본문은 저장하지 않는다.

| 이벤트 | 저장 상태 | 사용자 표시 |
| --- | --- | --- |
| 자동 확인 최신 | 성공 시각·앱 버전 갱신, 오류·available 제거 | 트레이 기본값, 설정 최신 |
| 자동 확인 새 버전 | 성공 시각·앱 버전·available 갱신 | 첫 발견 알림, 트레이 강조 |
| 같은 버전 자동 재확인 | 성공 시각 갱신 | 알림 반복 없음, 트레이 유지 |
| 더 높은 버전 발견 | available·lastNotified 갱신, snooze 해제 | 새 알림, 트레이 새 버전 |
| 자동 확인 실패 | 시도 시각·실패 횟수·짧은 오류 갱신 | 팝업 없음, 설정 오류 |
| 수동 확인 성공 | 성공 상태 갱신 | 최신이면 결과 안내, 새 버전이면 창 |
| `나중에` 또는 X | snooze 갱신 | 창 닫기, 트레이 강조 유지 |
| 설치 시작 | 런타임 설치 상태만 갱신 | 트레이·창 설치 중 |
| 설치 완료 | updater가 재시작 | 새 앱 첫 시작 확인으로 상태 정리 |

## 파일별 변경 범위

### `src-tauri/src/update.rs`

- 상태 스키마 v2와 기본값 추가
- `UpdateCheckSource`, `CheckEffects` 타입 추가
- 다음 확인 시각과 backoff를 계산하는 순수 함수 추가
- 자동·수동 실패 기록 분리
- 자동 알림과 창 표시 여부를 결정하는 작은 순수 함수 추가
- 최신 확인 시 오래된 `availableVersion` 정리
- 네트워크 확인이 끝난 뒤 상태를 다시 읽고 병합해 동시에 누른 `나중에` 상태를 덮어쓰지 않음
- v1 상태 마이그레이션 테스트 추가

### `src-tauri/src/lib.rs`

- `start_automatic_update_check`를 지속형 `start_update_monitor`로 교체
- `check_and_show_update`를 확인과 표시 적용 단계로 분리
- `check_update` `MenuItem` handle을 런타임 상태에 보관
- `MenuItem::set_text`로 트레이 문구 갱신
- 자동 발견은 알림+트레이만 적용하고 창을 열지 않음
- 트레이·설정 수동 확인은 기존 업데이트 창을 열 수 있게 유지
- 도달할 수 없는 `available && !should_open_window` 수동 알림 분기 제거
- 자동 확인 오류를 상태에 기록하되 반복 알림은 만들지 않음

### `src/ui/setup.js`와 `src/ui/setup.html`

- 초기 로드 때 `getUpdateState()`로 마지막 확인 결과 표시
- 최신, 업데이트 가능, 마지막 확인 실패, 아직 확인하지 않음을 구분
- 업데이트 가능 상태에서 버튼 문구를 `vX.Y.Z 업데이트 열기`로 변경

## 테스트 명세

### Rust 단위 테스트

- 상태 없음 -> 시작 15초 뒤 확인
- 같은 앱 버전, 성공 후 23시간 -> 남은 1시간 대기
- 같은 앱 버전, 성공 후 24시간 이상 -> 즉시 확인
- 앱 버전 변경 -> 이전 성공 시각과 무관하게 확인
- 앱 버전 변경 뒤 첫 자동 확인 실패 -> 즉시 반복하지 않고 15분 backoff
- 자동 실패 1회 -> 15분 뒤 재시도
- 자동 실패 2회 -> 1시간 뒤 재시도
- 자동 실패 3회 이상 -> 6시간 뒤 재시도
- 성공 -> 실패 횟수 초기화
- busy -> 실패 횟수 유지, 1분 뒤 재판단
- 수동 실패 -> 자동 실패 횟수 유지
- 같은 available 버전 -> 알림 반복 없음
- 더 높은 available 버전 -> snooze와 무관하게 발견 시 알림
- 최신 확인 -> 오래된 available 제거
- 저장된 available이 현재 앱 버전과 같거나 낮음 -> 트레이 기본 문구
- 네트워크 확인 중 `나중에` 상태 변경 -> 완료 저장이 해당 상태를 덮지 않음
- v1 JSON -> 필드 보존과 v2 기본값 적용
- 손상 JSON -> 기본 상태 복구

### UI·계약 테스트

- 앱 setup에서 `start_update_monitor`를 정확히 한 번 시작함
- updater plugin은 정확히 한 번 등록됨
- 프런트엔드 updater permission은 계속 없음
- 트레이 `check_update` ID가 유지됨
- 트레이 문구 네 상태가 계약에 포함됨
- Setup이 `getUpdateState()`와 수동 `checkForUpdate(true)`를 모두 사용함
- 자동 발견 경로에서 `show_window_by_label("update")`가 호출되지 않음
- 수동 발견 경로에서만 업데이트 창을 엶
- 업데이트 설치는 계속 버튼 클릭으로만 시작함

### Windows 수동 E2E

1. 기존 updater 회귀 검증으로 `1.2.2 -> 1.2.3` 사용자 승인 설치·재시작을 통과한다.
2. `1.2.3` 테스트 빌드와 v1 상태 파일을 준비한다.
3. 이전 성공 시각이 24시간 이내여도 앱 버전 변경을 감지해 시작 15초 뒤 한 번 확인하는지 본다.
4. 지속형 스케줄러 검증은 같은 `1.2.3`에서 마지막 성공 시각을 due 직전으로 준비하고, 테스트 manifest에 더 높은 테스트 버전을 공개한다.
5. 앱을 재시작하지 않고 due 시각에 알림과 트레이 문구가 바뀌는지 본다.
6. 알림을 닫아도 트레이 문구가 유지되는지 본다.
7. 트레이 항목을 눌러 업데이트 창, 버전과 릴리스 노트를 확인한다.
8. `나중에`를 누른 뒤 트레이 문구는 남고 알림은 반복되지 않는지 본다.
9. 더 높은 테스트 버전을 공개하면 기존 snooze보다 우선해 발견 시 새 알림이 뜨는지 본다.
10. 네트워크를 끊고 자동 backoff와 설정 오류 표시를 확인한다.
11. 다시 연결해 자동 또는 수동 성공 시 오류가 정리되는지 본다.
12. 테스트 업데이트 설치 뒤 앱 버전과 `~/.codex-usage-wrapper` 데이터 보존을 확인한다.

테스트 manifest는 공식 공개키·endpoint와 섞이지 않는 별도 설정과 테스트 키를 사용한다. 테스트를 위해 insecure transport나 서명 우회를 추가하지 않는다.

## 롤백 기준

다음 중 하나라도 발생하면 릴리스를 중단하고 one-shot 자동 확인으로 되돌린다.

- due가 아닌데 반복 네트워크 요청이 발생함
- 업데이트 창 또는 알림이 같은 버전에서 반복됨
- 트레이 메뉴가 응답하지 않거나 앱 종료를 막음
- 상태 파일 손상으로 앱 시작이 실패함
- 서명 검증 전 다운로드 파일이 실행됨
- 사용자 데이터가 삭제되거나 다른 Windows 계정 데이터와 섞임

롤백하더라도 기존 수동 `업데이트 확인`, 사용자 승인 설치, 서명 검증과 데이터 보존 경로는 유지한다.

## 설계 결정 요약

- 자동 모달 대신 알림+트레이를 선택한다.
- 알림 클릭은 현재 API 제약 때문에 완료 조건에서 제외한다.
- 앱이 켜진 동안 지속되는 스케줄러를 둔다.
- 성공 확인은 24시간, 실패는 15분/1시간/6시간 backoff를 사용한다.
- 앱 버전이 바뀌면 이전 cooldown을 우회한다.
- 알림·창 효과만 작은 순수 함수로 분리해 테스트한다.
- 사용자 승인 설치와 Tauri 서명 검증 계약은 변경하지 않는다.
