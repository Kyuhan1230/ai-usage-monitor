# Building a Windows quota forecaster for Codex CLI and Claude Code

Most AI coding usage tools answer a backward-looking question: how many tokens did I use?

That is useful, but it was not the question interrupting my longer coding sessions. I wanted to know:

> If I keep working at this pace, will my subscription quota run out before it resets?

That question led to **Codex Claude Usage**, a local Windows tray app for people who actively use Codex CLI, Claude Code, or both. It forecasts an exhaustion window, distinguishes a likely pre-reset failure from an ambiguous forecast, detects unusual daily usage, and turns the result into an action such as slowing down or switching models.

This article explains the engineering choices, including the parts that are deliberately conservative or still imperfect.

## Start with the decision, not the token count

The product is not an API billing dashboard. A subscription user usually cannot change the invoice generated by one more prompt, but they can change what they do before the next reset:

- postpone a large task;
- reduce the rate of use;
- inspect an accidental repetition loop;
- send a simple task to a lower-cost model;
- switch from one provider to the other.

That made three outputs more important than a large analytics surface:

1. an exhaustion range compared with the reset time;
2. confidence that explains how much history supports the range;
3. a concrete next action.

The data flow stays intentionally small:

```text
manual refresh or local activity
        |
        v
one-shot provider CLI collection
        |
        v
sanitized local quota history + numeric token totals
        |
        v
forecast range, anomaly checks, recommendations
        |
        v
on-demand WebView window -> destroyed when closed
```

## Keep authentication behind the provider CLI

The app does not read provider authentication tokens and does not copy browser cookies. It asks the already installed and authenticated CLI for the narrow result it needs.

For Codex, the collector starts the CLI app server, performs the initialization handshake, requests `account/rateLimits/read`, and stops the process. For Claude, the app can receive numeric quota fields through an optional `statusLine` hook and uses one `/usage` command when it needs an initial or fallback value.

Authentication probes run `codex login status` or `claude auth status`. The app uses only the exit result for the state decision and immediately discards command output because it can contain an account email or organization name.

There is a practical reason for this boundary beyond privacy: provider authentication formats are not an API contract for this project. Letting the official CLI own its credentials reduces the amount of sensitive and unstable behavior the app must duplicate.

## Do not turn “automatic” into an always-running CLI

Manual refresh is the default. Optional activity monitoring watches modification times of local Codex and Claude session files once per minute. New activity becomes a pending refresh, but collection cannot run more often than once every five minutes.

The provider commands are therefore short-lived:

- no background Codex or Claude process while the tray is idle;
- no local HTTP server;
- no port listening for browser traffic;
- no continuous quota polling.

The five-minute cooldown also prevents a burst of filesystem changes from spawning an equivalent burst of provider commands.

## Read token numbers without building a prompt archive

Quota history answers the forward-looking limit question. Local session JSONL supplies token counts for anomaly detection and approximate model comparisons.

The scanner extracts fields such as model, input tokens, cached tokens, and output tokens. It does not copy prompt text or response text into the analytics files. Claude message IDs are deduplicated, and a file metadata cache avoids reparsing unchanged session files on every scan.

This still has an explicit privacy boundary: the program reads local session files. The relevant guarantee is not “the files are never touched”; it is that prompt and response bodies are not copied into the derived analytics data.

## Forecast a range, then allow “unknown”

The forecast begins with the current quota cycle. If remaining quota falls from \(R_0\) to \(R_1\) over \(h\) hours, the cycle-average depletion rate is:

```text
rate = max(R0 - R1, 0) / h
expected exhaustion = latest sample time + R1 / rate
```

The app waits for at least five minutes of observed time and a positive depletion before producing a rate.

A single point estimate would look more precise than the evidence permits. The implementation therefore calculates rates over aggregated intervals within a limit-specific window, measures their median absolute deviation from the cycle average, and converts that variability into a forecast spread. The spread is bounded between 10% and 50% and narrows as more usable intervals accumulate.

That produces an earliest and latest exhaustion time:

- **risk** only when the entire forecast range ends before reset;
- **safe** only when the entire forecast range begins at or after reset;
- **unknown** when the range overlaps reset or the data is insufficient.

The overlap case matters. It prevents a colorful dashboard from turning weak evidence into a confident instruction.

## Make confidence inspectable

Confidence is based on observation time, actual depletion events, usable rate intervals, and forecast spread:

| Confidence | Minimum evidence |
| --- | --- |
| High | 6 observed hours, 5 depletion events, 5 rate intervals, spread at most 25% |
| Medium | 2 observed hours, 2 depletion events, 3 rate intervals, spread at most 50% |
| Low | Anything below those thresholds |

The UI shows sample count, observed duration, interval count, spread, and the source collection time. A low-confidence forecast remains visible as a reference, but it does not trigger the same slowdown recommendation as a medium- or high-confidence risk.

This is a heuristic model, not a guarantee about provider-side enforcement. Its purpose is to expose the evidence behind a decision rather than conceal uncertainty.

## Compare today with the user’s own recent baseline

A fixed “large token count” threshold behaves badly across different users. The anomaly detector instead looks at the previous seven active days for each provider.

It requires at least three non-zero historical days, then calculates:

```text
baseline = median(previous active days)
threshold = max(baseline * 1.8, baseline + 3 * MAD)
```

Today is flagged only when usage is above that threshold and at least 10,000 tokens. The result reports the multiplier over the recent median, which is easier to act on than a generic severity score.

This detector is intentionally simple. It will not distinguish a planned benchmark run from an accidental automation loop; it tells the user that today is unusual and asks them to inspect the cause.

## Recommendations are derived, not generated

The app does not send local analytics to an LLM to write advice. Recommendations come from deterministic rules:

- at 10% remaining or below, defer large work until reset;
- for a non-low-confidence pre-reset risk, compare the current rate with `remaining / hours until reset`;
- round the required reduction upward to the next 5%;
- for a token spike, inspect automated repetition and unusually large context;
- show a lower-cost model comparison only when the same-token estimate saves at least 20% and USD 0.01.

Cost figures are API list-price equivalents. They are not claims about the user’s subscription invoice.

## Destroy the WebView instead of hiding it

Tauri keeps the Rust tray process small, but a Windows WebView2 window is not small merely because the HTML is simple.

The measured release executable is 4.41 MB. Cold tray idle used 11.43 MB with no WebView process. Opening the full UI used about 427 MB across the app and WebView2 processes.

The last number is uncomfortable, so it is published rather than averaged away. The mitigation is lifecycle-based: secondary windows are created on demand, and closing a window destroys it. Hiding the window would keep the WebView cost resident while providing no value.

The app also creates a WebView from a worker because constructing it directly inside a synchronous Tauri command or tray event can deadlock on Windows.

## Unsigned Windows software has a trust problem

The current installer is not Authenticode-signed. The SignPath Foundation application was declined on 2026-07-23 because the project does not yet have enough external adoption and independent references.

That makes installation trust part of the product work:

- the GitHub Release publishes the exact SHA-256 digest;
- release notes distinguish the Tauri updater signature from a Windows publisher signature;
- SmartScreen’s **Unknown publisher** warning is disclosed before download;
- source-build instructions are available;
- VirusTotal is treated as another signal, never as a safety certificate.

The correct next step is not to hide the warning. It is to earn enough real usage, feedback, and external references to reapply with stronger evidence.

## What is still weak

The app currently has several important limitations:

- Windows only;
- forecasts require history and can remain unknown;
- provider CLI output can change;
- model price comparisons do not represent subscription billing;
- local session schemas can drift;
- the installer remains unsigned;
- the memory cost is high while WebView2 is open.

Those limitations define the beta questions better than another feature list does. The most useful reports now are:

1. Did SmartScreen stop the install?
2. Did the first provider refresh succeed?
3. Was the forecast range and confidence understandable?
4. Did the recommendation change the next action?
5. What was the single biggest point of friction?
